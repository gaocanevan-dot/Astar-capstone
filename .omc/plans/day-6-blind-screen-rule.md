# Day-6 — Blind Screen Rule (R2 — gate condition for Step 0a)

**Purpose**: This file MUST be committed to git BEFORE
`scripts/day6_blind_screen.py` (Step 0a) runs. It codifies the
operational rules that protect the Day-6 sweep against:
- Audit-as-intervention scope creep (Architect v1 + v5 closures)
- Ad-hoc Jaccard threshold tuning (Critic v6 closure)
- Post-hoc narrative authoring (Day-5b discipline)

The mechanical scripts read AC10d threshold from this file
(`scripts/day6_audit.py:read_ac10d_threshold`). Editing the threshold
here AFTER Step 0a runs is a stop-rule violation.

---

## Rule 1 — Blind buildability pre-screen

`scripts/day6_blind_screen.py` records ONLY the aggregate `buildable_count`
integer. Per-ID build success/failure is NOT logged.

**Operator behaviour during Step 0a**:
- Run the script. Read the output count. Make the routing decision.
- DO NOT inspect which specific cases failed.
- DO NOT attempt per-case toolchain fixup ("just fix the foundry config
  for that one case") — this would convert the audit into the actual
  intervention.
- If buildable < 18, escalate to Path B (Step 0c spot-check). Do NOT
  shrink n_target below 15.

**Why this rule exists**: in v1 review, Architect identified
"audit-as-intervention" risk where the act of auditing the corpus would
become the de facto intervention being measured. v5 Critic confirmed the
risk applies symmetrically to buildability. The blind-count rule is the
mechanical defence.

## Rule 2 — AC10d Jaccard threshold (R1 calibrated)

**AC10d threshold: 0.400**

Calibration source: `data/evaluation/day6_jaccard_distribution.md`
(generated by `scripts/calibrate_jaccard.py` against the 23 clean ACF
candidates × 85 anti_patterns library entries).

### Calibration summary

| statistic | value |
|---|---|
| P50 max-Jaccard | 0.200 |
| P75 max-Jaccard | 0.400 |
| P90 max-Jaccard | 0.642 |
| P95 max-Jaccard | 0.667 |
| max | 1.000 (ACF-096 is a duplicate of ACF-081) |
| natural gap (≥0.10 wide) | 0.833 (between 0.667 and 1.000) |
| **contamination hub** | **ACF-078 is top-1 match for 7 holdout cases at Jaccard 0.375-0.667** |

### Why threshold = 0.400 (not natural gap 0.833)

The natural-gap method picks 0.833 because that's the largest gap
between adjacent values. But this misses a substantial **contamination
hub**: 7 of the 23 holdout cases (ACF-111, 112, 113, 115, 116, 117, 118)
all max-match against the SAME library entry (ACF-078) with Jaccard
0.375 to 0.667. These are sequentially-numbered DeFi-Hack cases that
share attack structure with the curated library's ACF-078 entry.

If we used 0.833 threshold, only ACF-096 (the perfect duplicate at 1.000)
would drop, and 7 hub-coupled cases would survive into the holdout. RAG
retrieval@3 on those 7 cases would look fake-good (system retrieves
ACF-078, which has 50%+ token overlap with the holdout's vulnerable
code). This would inflate AC11 and AC12 spuriously.

**Threshold 0.400 is chosen specifically to catch the hub cluster**
while not over-rejecting legitimately distinct cases. It corresponds
to P75 of the empirical distribution and to the natural separation
between hub-affected cases (Jaccard ≥ 0.40 against ACF-078) and
clean cases (max-Jaccard ≤ 0.318 against various library entries).

### Predicted impact

At threshold 0.400, AC10d will drop:
- ACF-096 (1.000, matches ACF-081 — perfect duplicate)
- ACF-115 (0.667, matches ACF-078)
- ACF-117 (0.667, matches ACF-078)
- ACF-116 (0.545, matches ACF-078)
- ACF-112 (0.500, matches ACF-078)
- ACF-113 (0.400, matches ACF-078)
- ACF-118 (0.400, matches ACF-078)

→ 7 cases dropped, 16 remain.

If forge buildability of these 16 is ≥18, n=15 routing applies (after
3 calibration cases reserved). If <18 buildable from the 16-case
post-Jaccard pool, Step 0c spot-check on Path B fires immediately.

**Realistic forecast**: with only 16 candidates after Jaccard, n=18
routing is NOT achievable. Best case is n=15 routing if 18+ build.
Most likely outcome is Path B activation, in which case the Day-6
RAG mechanism is NOT evaluated this round and the architectural work
(`src/agent/rag/`) is preserved for re-use on the next viable corpus.

## Rule 5 — Modular RAG architecture commitment

The Day-6 implementation lives in `src/agent/rag/` (NOT inside
`src/agent/react/`). This separation is binding:

- `loop.py` imports only from `agent.rag` package (interface + registry).
- New Day-7+ injectors land as new `RAGInjector` subclasses in `agent.rag`.
- Loop never reaches into a concrete injector's internals.

This rule preserves the ablation pathway. Violations (e.g. inlining
RAG logic into `loop.py`) are reverted before merge.

## Rule 6 — Stop rule and pool exhaustion

Commit messages for any sweep on the 23-case clean ACF pool MUST
include literal `LAST_POSTHOC_ROUND_ON_HOLDOUT_V1`.

After the first sweep scores these 23 IDs, they cannot be used as
fresh validation evidence again. Characterization runs (e.g. swapping
embedding model, expanding library) MAY reuse the same IDs but their
results must be tagged as `characterization`, never as
`validation`. This is the v6 narrowed pool-exhaustion clause.
